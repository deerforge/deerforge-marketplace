DeerFlow Security Hardening Guide

Lock Down Your DeerFlow Deployment for Production

Version 1.0 • March 2026 • DeerForge

Who This Guide Is For

You've got DeerFlow running. Maybe it's on a DigitalOcean droplet, a VPS, or a home server. It works. Io is responding. You're building.

But right now, anyone who finds your IP address can:

  • Send unlimited prompts to your AI agent
  • Run up your Anthropic API bill without your knowledge
  • Execute GitHub operations using your Personal Access Token
  • Access your entire DeerFlow interface with no credentials

This guide fixes that. Every step in this guide was executed on a live DeerFlow 2.0 deployment on Ubuntu 24.04 LTS. The commands are exact. The gotchas are real.

What you'll have when you're done:

  • UFW firewall blocking all ports except SSH and your DeerFlow UI
  • nginx basic authentication requiring credentials to access the UI
  • Internal Docker ports confirmed isolated from the public internet
  • Telegram bot (if you use one) authenticated through nginx
  • External verification that everything actually works

Time required: 45–60 minutes. Budget extra time if you hit the Docker/UFW gotcha or the nginx auth debugging steps — those are the two places most likely to need iteration.

Before You Start

Prerequisites

  • DeerFlow 2.0 running on Ubuntu 24.04 LTS
  • SSH access to your server
  • A password manager (you'll be creating credentials)
  • WinSCP or another SFTP client (needed for one file edit — terminal paste breaks on large configs)

Critical: Open Two SSH Terminals

Before touching anything, open two separate SSH sessions to your server. Keep both open throughout this entire guide.

If you accidentally lock yourself out of SSH, terminal 2 saves you. If terminal 2 is also broken, you need your hosting provider's browser console. Know where that is before you start.

ssh -i /path/to/your/key root@YOUR_SERVER_IP

Do this twice. Confirm both terminals show the server prompt before proceeding.

The Recovery Plan

If SSH breaks: log into your hosting provider's dashboard and use the browser-based console to access your server. For DigitalOcean, this is Droplets → your droplet → Console.

Step 1: UFW Firewall

What We're Doing

Enabling a firewall that blocks all inbound traffic except SSH (port 22) and your DeerFlow UI (port 2026). Everything else — including any ports that might accidentally be exposed — gets blocked by default.

Check Current Status

ufw status

Expected output on a fresh deployment:

Status: inactive

Allow SSH First — This Order Matters

Do not enable the firewall before allowing SSH. If you do, you will lock yourself out.

ufw allow 22/tcp

Expected output:

Rules updated
Rules updated (v6)

Allow Your DeerFlow Port

ufw allow 2026/tcp

Expected output:

Rules updated
Rules updated (v6)

Enable the Firewall

ufw enable

You'll see a warning about disrupting SSH connections. Type y and press enter.

Command may disrupt existing ssh connections. Proceed with operation (y|n)? y
Firewall is active and enabled on system startup

Immediately switch to terminal 2 and confirm you're still connected:

echo "still connected"

If terminal 2 responds, you're good. If not, use the browser console to fix it.

Verify Clean Ruleset

ufw status verbose

Expected output:

Status: active
Logging: on (low)
Default: deny (incoming), allow (outgoing), deny (routed)

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW IN    Anywhere
2026/tcp                   ALLOW IN    Anywhere
22/tcp (v6)                ALLOW IN    Anywhere (v6)
2026/tcp (v6)              ALLOW IN    Anywhere (v6)

If you see duplicate entries (e.g., 2026 and 2026/tcp both listed), clean them up:

ufw delete allow 2026

This removes the non-specific entry, keeping only the explicit 2026/tcp rule.

Step 2: nginx Installation & Basic Auth

What We're Doing

Installing nginx as a reverse proxy with HTTP Basic Authentication. This puts a username/password prompt in front of your DeerFlow UI while keeping the internal Docker setup unchanged.

Install nginx

apt update && apt install nginx apache2-utils -y

Create Authentication Credentials

Replace your_username with your preferred login name:

htpasswd -c /etc/nginx/.htpasswd your_username

You'll be prompted to enter a password twice. Use a strong password and save it in your password manager.

Configure nginx

Create the configuration file:

nano /etc/nginx/sites-available/deerflow

Add this configuration (replace YOUR_SERVER_IP with your actual server IP):

server {
    listen 2026;
    server_name YOUR_SERVER_IP;

    location / {
        auth_basic "DeerFlow Access";
        auth_basic_user_file /etc/nginx/.htpasswd;
        
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
        proxy_read_timeout 86400;
    }
}

Enable the Configuration

ln -s /etc/nginx/sites-available/deerflow /etc/nginx/sites-enabled/
nginx -t
systemctl restart nginx

The nginx -t command should return "syntax is ok" and "test is successful".

Step 3: Stop DeerFlow Direct Access

What We're Doing

Modifying your DeerFlow Docker configuration to only accept connections from localhost, not from the public internet. This forces all traffic through nginx (and its authentication).

Find Your docker-compose.yml

find / -name "docker-compose.yml" -type f 2>/dev/null | grep -E "(deerflow|claude)"

Common locations:

  • /root/deerflow/docker-compose.yml
  • /opt/deerflow/docker-compose.yml
  • /home/ubuntu/deerflow/docker-compose.yml

Edit the Docker Configuration

Open your docker-compose.yml file. Look for the ports section under the main service. It probably looks like:

ports:
  - "2026:3000"

Change it to:

ports:
  - "127.0.0.1:3000:3000"

This binds the Docker container to localhost only, not to all network interfaces.

Restart DeerFlow

Navigate to your DeerFlow directory and restart:

cd /path/to/your/deerflow/directory
docker-compose down
docker-compose up -d

Verify the Change

docker ps

You should see something like:

PORTS
127.0.0.1:3000->3000/tcp

If you see 0.0.0.0:2026->3000/tcp, the change didn't take effect. Double-check your docker-compose.yml edit.

Step 4: Verification

Test External Access

From your local machine (not the server), test that direct access is blocked:

curl http://YOUR_SERVER_IP:3000

This should fail or timeout. If it returns DeerFlow content, your Docker configuration isn't correct.

Test nginx Authentication

This should prompt for credentials:

curl http://YOUR_SERVER_IP:2026

Expected response:

<html>
<head><title>401 Authorization Required</title></head>
<body>
<center><h1>401 Authorization Required</h1></center>
<hr><center>nginx/1.18.0 (Ubuntu)</center>
</body>
</html>

Test Authenticated Access

curl -u your_username:your_password http://YOUR_SERVER_IP:2026

This should return the DeerFlow interface HTML.

Browser Test

Open http://YOUR_SERVER_IP:2026 in your browser. You should see an HTTP Basic Auth dialog. Enter your credentials. DeerFlow should load normally.

Troubleshooting

nginx Won't Start

Check the error log:

tail -f /var/log/nginx/error.log

Common issues:

  • Port already in use: Something else is using port 2026. Check with netstat -tlnp | grep 2026
  • Syntax error: Run nginx -t to check your configuration

Authentication Not Working

Verify the htpasswd file exists and has content:

ls -la /etc/nginx/.htpasswd
cat /etc/nginx/.htpasswd

The file should contain a line like:

your_username:$apr1$hash...

Docker Container Not Accessible

Check if the container is running:

docker ps

Check container logs:

docker logs container_name

Verify nginx can reach the container:

curl http://localhost:3000

This should return DeerFlow content when run from the server.

UFW Blocking Everything

If you can't access anything, use the browser console to disable UFW:

ufw disable

Then re-enable it step by step, testing access after each rule.

Optional: Telegram Bot Security

If you're using DeerFlow's Telegram integration, you need to secure the webhook endpoint too.

Add Telegram Route to nginx

Edit your nginx configuration:

nano /etc/nginx/sites-available/deerflow

Add this location block inside the server block:

location /telegram {
    auth_basic "Telegram Webhook";
    auth_basic_user_file /etc/nginx/.htpasswd;
    
    proxy_pass http://localhost:3000/telegram;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

Update Telegram Webhook URL

In your Telegram bot configuration, update the webhook URL to:

http://username:password@YOUR_SERVER_IP:2026/telegram

Replace username and password with your nginx credentials.

Test Telegram Integration

Send a message to your bot. It should work normally, but now the webhook is protected by authentication.

Security Checklist

After completing this guide, verify each item:

Firewall Status

  • ✓ UFW is active and enabled
  • ✓ Only ports 22 (SSH) and 2026 (DeerFlow) are allowed
  • ✓ Default policy denies all other inbound traffic

Authentication

  • ✓ DeerFlow UI requires username/password
  • ✓ Direct access to port 3000 is blocked
  • ✓ nginx basic auth is working correctly

Docker Security

  • ✓ Docker containers only bind to localhost
  • ✓ No Docker ports exposed to public internet
  • ✓ All traffic goes through nginx proxy

External Verification

  • ✓ Port scan shows only 22 and 2026 open
  • ✓ Unauthenticated requests return 401
  • ✓ Authenticated access works normally

Operational Security

  • ✓ Strong passwords stored in password manager
  • ✓ SSH keys properly secured
  • ✓ Regular security updates scheduled
  • ✓ Backup and recovery plan documented

Maintenance

Regular Updates

Keep your system updated:

apt update && apt upgrade -y

Monitor nginx Logs

Check for suspicious access attempts:

tail -f /var/log/nginx/access.log

Backup Configuration

Backup your nginx configuration:

cp /etc/nginx/sites-available/deerflow ~/deerflow-nginx-backup.conf
cp /etc/nginx/.htpasswd ~/deerflow-auth-backup

UFW Log Monitoring

Check blocked connection attempts:

grep "UFW BLOCK" /var/log/ufw.log

Adding New Users

To add additional users to the authentication system:

htpasswd /etc/nginx/.htpasswd new_username

Restart nginx after adding users:

systemctl restart nginx

What's Next

Your DeerFlow deployment is now secured with basic authentication and firewall protection. This addresses the most critical security gaps in a default installation.

Additional Security Measures

For production deployments handling sensitive data, consider:

  • SSL/TLS certificates: Use Let's Encrypt for HTTPS
  • Fail2ban: Automatic IP blocking for brute force attempts
  • VPN access: Restrict access to specific IP ranges
  • Database encryption: If storing sensitive conversation history
  • API key rotation: Regular rotation of Anthropic and GitHub tokens
  • Container hardening: Run containers as non-root users
  • Network segmentation: Isolate DeerFlow in a separate subnet

Monitoring and Alerting

Set up monitoring for:

  • Failed authentication attempts
  • Unusual API usage patterns
  • System resource consumption
  • Container health and uptime

Documentation

Document your specific configuration:

  • Server IP addresses and ports
  • Authentication credentials (in password manager)
  • Backup and recovery procedures
  • Emergency contact information

Remember: Security is an ongoing process, not a one-time setup. Regularly review and update your security measures as your deployment grows.
